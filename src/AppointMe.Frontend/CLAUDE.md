# CLAUDE.md — AppointMe Frontend

Guidance for Claude Code (claude.ai/code) when working with the AppointMe React/TypeScript frontend. This file is scoped to `src/AppointMe.Frontend/`. The repo root has its own `CLAUDE.md` for backend conventions — read both when a task spans the stack.

## Build and Development Commands

```bash
npm run dev               # Vite dev server on https://localhost:5173 (mkcert auto-HTTPS)
npm run build             # tsc -b + Vite production build
npm run lint              # ESLint over src/
npm run generate:api      # Regenerate API client from backend OpenAPI (needs Aspire up)
```

**Type-check from CI/scripts**:
```bash
npx tsc --noEmit --ignoreDeprecations 6.0
```
The `--ignoreDeprecations 6.0` flag silences a pre-existing `baseUrl` deprecation warning in `tsconfig.json`. Without it, `tsc --noEmit` fails on that warning even when there are no actual type errors.

**API regeneration**: `npm run generate:api` calls orval which fetches `https://localhost:7233/openapi/v1.json`. The backend must be running via Aspire (`cd src/AppointMe.Aspire && dotnet run` from the repo root). `NODE_TLS_REJECT_UNAUTHORIZED=0` is set in the script because mkcert certs are self-signed.

## Architecture

### Folder layout

```
src/
├── api/                # Generated by orval — DO NOT hand-edit
│   ├── appointme.ts          # query/mutation hooks
│   └── appointme.schemas.ts  # TypeScript types
├── app/                # Route-level pages and layouts
│   ├── <feature>/            # Vertical slice per feature (appointments, customers, team, …)
│   │   ├── <feature>.tsx     # page entry (route component)
│   │   ├── index.ts          # barrel — exports the page
│   │   └── <action>/         # one folder per user action (verb-first)
│   │       ├── <action>-dialog.tsx
│   │       └── index.ts
│   ├── layouts/, router/, providers/, auth/, onboarding/, not-found/
├── components/
│   ├── ui/             # shadcn primitives (vendored, treat as semi-third-party)
│   ├── format/         # Date/time/timezone system (see below)
│   ├── auth/           # current-user / current-company contexts, <Can>
│   ├── modal-dialog/   # programmatic modal system
│   ├── error/, onboarding/, …
├── hooks/              # global hooks (use-mobile.ts, …)
├── lib/
│   ├── axios.ts        # axios instance + X-Company-Id interceptor + 401 redirect
│   ├── utils.ts        # cn() — clsx + tailwind-merge
│   └── query-client.ts
└── assets/             # static images
```

### Vertical slice convention (mirrors the backend)

Each feature folder under `app/` has the route page at its root and one sub-folder per **user action**. Action folders are verb-first (`schedule-appointment/`, `reschedule-appointment/`, `cancel-appointment/`). Read-view concerns (the calendar) live alongside the page as their own slice folder (`calendar/`). See `app/appointments/` for the canonical example.

Slice contents typically include: the dialog/form component, its zod schema (co-located), any slice-private hooks. Cross-slice helpers stay at the feature root, not inside any one slice.

### Component split rule (fast refresh)

ESLint enforces `react-refresh/only-export-components`. A `.tsx` file must export **only** React components. Hooks, types, constants, and plain functions go in sibling `.ts` files. Pattern in use:

- `formatted-date.tsx` (component) + `use-formatted-date.ts` (hook + `DateFormat` type + internal formatters).
- `calendar.tsx` (component) + `calendar-config.ts` (`ViewKey`, `VIEW_MAP`).

If you add a helper to a `.tsx` file the lint warning fires and HMR drops to full reload.

## Key Patterns

### Data fetching — orval + TanStack Query

All API calls go through orval-generated hooks in `src/api/appointme.ts`:

```ts
const { data: appointments } = useGetAppointments(dateRange);
const { mutateAsync: schedule } = useScheduleAppointment();
await schedule({ data: { ... } });
await queryClient.invalidateQueries({ queryKey: getGetAppointmentsQueryKey(dateRange) });
```

- Queries use `useSuspenseQuery` (orval config); route pages must sit inside `<Suspense>` (handled by `app/router/`).
- The axios mutator (`lib/axios.ts`) injects `X-Company-Id` from `currentCompanyStore` on every request and redirects to `/auth/login` on 401.
- All API datetimes are **UTC ISO strings**. Convert to/from company-tz wall clock for display and inputs (see Date/Time below).

### Modal dialogs — programmatic API

Dialogs are opened from code, not declared inline:

```tsx
const modalDialog = useModalDialog();
const ok = await modalDialog.open<boolean>(props => (
    <ScheduleAppointmentDialog {...props} initialDate={...} />
));
```

The dialog component receives `{ resolve, visible }` from `IModalProps<T>`. It controls itself via `visible` and calls `resolve(value)` to close.

**Radix `<Dialog>` controlled-mode quirk**: every programmatically-opened dialog needs a hidden trigger to silence Radix's "missing required child" warning:

```tsx
<Dialog open={visible} onOpenChange={open => !open && resolve(false)}>
    <DialogTrigger asChild>
        <button type='button' aria-hidden tabIndex={-1} className='hidden' />
    </DialogTrigger>
    <DialogContent>…</DialogContent>
</Dialog>
```

**Radix `<Popover>` positioned at click coords**: use `<PopoverTrigger asChild>` wrapping a `pointer-events-none fixed` 1×1 `<span>` — Radix uses it as both trigger and anchor. See `app/appointments/calendar/appointment-preview.tsx`.

### Forms — react-hook-form + zod + shadcn Field

```tsx
const formSchema = z.object({
    date: z.string().min(1, 'Date is required'),
    notes: z.string().max(NOTES_MAX_LENGTH, '…').optional().nullable(),
});
type FormValues = z.infer<typeof formSchema>;

const { control, formState, handleSubmit } = useForm<FormValues>({
    resolver: zodResolver(formSchema),
    defaultValues: { … },
});

<Controller
    name='date'
    control={control}
    render={({ field, fieldState }) => (
        <Field data-invalid={fieldState.invalid}>
            <FieldLabel htmlFor='date'>Date</FieldLabel>
            <Input id='date' {...field} type='date' />
            {fieldState.error && <FieldError errors={[fieldState.error]} />}
        </Field>
    )}
/>
```

Zod schema is **co-located** with the dialog, not in a separate file. Submit buttons live in `<DialogFooter>` and submit the form via `form="<form-id>"` attribute (form sits inside `<DialogContent>`).

### Auth and permissions

- `useCurrentUser()` → full `GetCurrentUserResponse` (auth state, memberships).
- `useCurrentCompany()` → current `CompanyMembership` (`companyId`, `companyName`, `timeZone`).
- `<Can permission='appointments:schedule'>…</Can>` gates UI by permission key. Permission strings are colon-namespaced (`<resource>:<action>`).

### Date / time / timezone — `components/format/`

**Core rule**: API datetimes are UTC ISO. All display and form inputs render in the **company's IANA timezone**, read from `useTimeZone()`. **No `date-fns`** — built-in `Intl` only.

```tsx
// Single date / time — renders <time dateTime=…>
<FormattedDate date={appointment.start} format='shortNoYear' />
<FormattedDate date={appointment.start} format='time' />

// Date range — renders <span>
<FormattedDateRange from={range.from} to={range.to} />

// Non-JSX call sites
const formatDate = useFormattedDate();
const formatRange = useFormattedDateRange();
const timeZone = useTimeZone();

// HTML form inputs
toDateInputValue(iso, tz)              // "2026-03-05" for <input type="date">
toTimeInputValue(iso, tz)              // "14:30"      for <input type="time">
parseInputDateTime(date, time, tz)     // company-tz wall clock → UTC ISO
```

`DateFormat` values: `short`, `shortWithWeekday`, `shortNoYear`, `full`, `monthDay`, `month`, `weekday`, `year`, `time`, `dateTime`, `numericDate`.

`<Calendar>` (the FullCalendar wrapper) takes a `timeZone` prop; without it the grid renders in browser-local time. The page does `const timeZone = useTimeZone()` and threads it down.

### Routing

React Router v7. Layout shell at `app/router/app-shell.tsx`, route table at `app/router/router.tsx`. Breadcrumbs and active-nav state come from route `handle` via `useBreadcrumbs()` / `useActiveNav()`. Protected routes are nested under `<CurrentCompanyProvider>` + `<UserAccessProvider>` + `<ModalDialogProvider>`.

### Styling

Tailwind CSS 4 via `@tailwindcss/vite`. Always use shadcn theme tokens (`bg-background`, `text-foreground`, `border-border`, `text-muted-foreground`, `text-destructive`, `bg-primary/5`, …) — never raw hex colours. Combine class names via `cn()` from `@/lib/utils`.

## Naming Conventions

- **Filenames**: kebab-case everywhere (`schedule-appointment-dialog.tsx`, `use-formatted-date.ts`).
- **Components**: PascalCase exports.
- **Hooks**: `useX` exports, `use-x.ts` filenames.
- **Index barrels**: every feature/slice folder has an `index.ts` that exports its public surface.
- **Permission keys**: `<resource>:<action>` (e.g., `appointments:schedule`).
- **Indentation**: 4 spaces. Quotes: single (incl. JSX). Trailing commas everywhere. Print width 120. Enforced by Prettier with `@trivago/prettier-plugin-sort-imports` and `prettier-plugin-tailwindcss`.

## Project-specific gotchas

- **Don't hand-edit `src/api/appointme.ts` or `src/api/appointme.schemas.ts`** — they're regenerated by orval with `clean: true`, which overwrites everything. If the backend changes shape, run `npm run generate:api` (Aspire must be running).
- **Don't import `date-fns`** — removed from the project. Use `Intl` and the `components/format/` helpers.
- **Don't use the browser's local timezone for appointment display** — always go through `useTimeZone()` / `<FormattedDate>` / `parseInputDateTime`. The browser tz is only used as a fallback when no company context is loaded yet.
- **Don't mix component and non-component exports in a `.tsx` file** — ESLint's `react-refresh/only-export-components` warns and HMR breaks. Split helpers into a sibling `.ts` file.
- **Don't render Radix `<Dialog>` or `<Popover>` controlled-mode without a Trigger** — Radix logs a warning. Use the hidden-trigger pattern shown above.
- **Don't use raw hex colours** — use Tailwind theme tokens so dark mode / theme changes propagate.
- **Don't put zod schemas in shared files** — co-locate with the dialog/form that uses them.
